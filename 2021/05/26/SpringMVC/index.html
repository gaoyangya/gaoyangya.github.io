1.回顾MVC

1.1什么是MVC

  • MVC是模型(Model)、视图(View)、控制器(Controller)的简写，是一种软件设计规范。
  • 是将业务逻辑、数据、显示分离的方法来组织代码。
  • MVC主要作用是降低了视图与业务逻辑间的双向偶合。
  • MVC不是一种设计模式，MVC是一种架构模式。当然不同的MVC存在差异。

Model（模型）：数据模型，提供要展示的数据，因此包含数据和行为，可以认为是领域模型或JavaBean组件（包含数据和行为），不过现在一般都分离开来：Value Object（数据Dao） 和 服务层（行为Service）。也就是模型提供了模型数据查询和模型数据的状态更新等功能，包括数据和业务。

View（视图）：负责进行模型的展示，一般就是我们见到的用户界面，客户想看到的东西。

Controller（控制器）：接收用户请求，委托给模型进行处理（状态改变），处理完毕后把返回的模型数据返回给视图，由视图负责展示。也就是说控制器做了个调度员的工作。

最典型的MVC就是JSP + servlet + javabean的模式。

1.2MVC框架要做哪些事情

  1. 将url映射到java类或java类的方法 .
  2. 封装用户提交的数据 .
  3. 处理请求–调用相关的业务处理–封装响应数据 .
  4. 将响应的数据进行渲染 . jsp / html 等表示层数据 .

2.什么是SpringMVC

2.1 SpringMVC简介

  • SpringMVC也叫Spring web mvc,说明它是spring的核心技术,是Spring框架的一部分(也可以单独理解为一个框架),是在spring3.0之后发布的
  • SpringMVC是一个容器,管理对象的,使用ioc核心技术.SpringMVC管理界面层中的控制器对象(最早用的是Servlet)
  • SpringMVC底层也是Servlet,是以Servlet为核心,接受请求,处理请求,显示处理结果给用户
  • 处理用户的请求:
    用户发起请求===SpringMVC(就相当于一个Servlet嘛)===spring(使用业务方法中的service)===mybatis===mysql数据库

2.2SpringMVC中的核心Servlet—DispatcherServlet

  • DispatcherServlet是框架的一个servlet对象.负责接受请求,响应处理结果
  • DispatcherServlet叫做中央调度器，父类是HttpServlet
  • DispatcherServlet也叫作前端控制器(front controller)

SpringMVC是管理控制器对象,原来没有SpringMVC之前使用servlet作为控制器对象使用(可以把SpringMVC理解成servlet的一个升级).现在通过SpringMVC容器创建一种叫做控制器的对象,代替servlet行使控制器的功能

SpringMVC主要使用注解的方式,创建控制器对象,处理请求

所谓SpringMVC的注解式开发是指:在代码中通过对类与方法的注解,便可完成处理器在SpringMVC容器中的注册

2.3 SpringMVC优点

  1. 基于MVC架构
    基于MVC架构,功能分工明确,解耦合
  2. 容易理解,上手快;使用简单
    就可以开发一个注解的SpringMVC项目,SpringMVC也是轻量级的,jar很小,不依赖特定的接口和类
  3. 作为Spring框架的一部分,能够使用spring的ioc和aop,方便整合其他框架

3.6 创建一个普通的类,作为控制器使用(代替之前servlet)

1)在类的上面加@Controller注解

​ @Controller: 创建控制器(处理器)对象

​ 控制器:叫做后端控制器(back controller),自定义的类处理请求的
​ 位置:在类的上面,表示创建此类的对象,对象放在SpringMVC的容器中

2)在类中定义方法,方法的上面加上@RequestMapping注解
方法处理请求的,相当于servlet的dopost,doGet

定义方法,处理请求    以前要用servlet重写doGet(),doPost()
现在自定义方法来处理请求

SpringMVC框架,使用控制器类中的方法,处理请求
格式:      public ModelAndView 方法名(形参){}
使用这个方法处理请求,能处理请求的方法叫做控制器方法
方法特点:
1.方法的形参,表示请求中的参数
2.方法的返回值,表示本次请求的处理结果(数据和视图)
        Model:表示数据
        View:表示视图
@RequestMapping:请求映射
       属性:value 请求中的uri地址,唯一值,以"/"开头
       位置:1.在方法的上面(必须)
           2.在类定义的上面(可选)
       作用:把指定的请求,交给指定的方法处理,等同于url-pattern

@Controller
public class MyController  {
    @RequestMapping(value = "/some.do")
    public ModelAndView dosome(){
        //调用service对象,处理请求,返回数据
        ModelAndView mv=new ModelAndView();
        //添加数据
        mv.addObject("msg", "处理了some.do请求");
        mv.addObject("fun","执行了dosome方法");
        //通过视图将数据显示给用户=>显示,指定视图setViewName("视图的完整路径")
        mv.setViewName("/show.jsp");//斜杠"/"是根的意思,也就相当于webapp
        //返回结果
        return  mv;
    }
}

  • 当框架调用完dosome()方法后,得到返回值ModelAndView

  • 框架会在后序的处理逻辑中,处理mv对象里的数据和视图

  • 对数据执行request.setAttribute(“msg”, “处理了some.do请求”);把数据放入到request作用域

  • 对视图执行forward转发操作,等同于request.getRequestDispatcher(“/show.jsp”).forward(..)

4.1 简单的处理过程

  1. 用户发起了some.do
  2. Tomcat接受了请求,根据web.xml里<url-pattern>里的*.do将some.do请求交给了<servlet-name>里的myweb这个中央调度器
  3. DispatcherServlet(中央调度器)本身不处理请求,最终是MyController.java类里的代码片段干的活.
    中央调度器把活分配给MyController.java最终返回mv对象
  4. mv显示给用户

用户some.do———–>DispatcherServlet————->MyController

5. 视图文件受保护

问题:Tomcat启动起来之后,在index.jsp所对应的浏览器页面(第一个springMVC)发some.do请求,为什么some.do请求能够交给中央调度器处理?

  1. Tomcat比对web.xml里<url-pattern>
  2. 交给<servlet-name>处理
  3. 接受了some.do的请求,程序中有中央调度器,中央调度器读了<init-param>里<param-init>的配置文件
  4. some.do请求的地址和MyController.java的@RequestMapping里的value值对应

Tomcat启动之后,按理说show.jsp应该是从index.jsp跳转过去的,但是如果地址栏直接输入:

==>没有数据,请求无效

所以应该让页面收到某种保护,不应该直接访问到

===>把受保护的视图文件扔到WEB-INF里面去

因为WEB-INF是一个受保护的文件夹,里面的资源从浏览器是访问不了的,但是在Java程序中是可以访问的

为了方便管理,一般是在WEB-INF下面专门建一个文件,存放视图文件

提醒:路径也变了,要在前面加上”/WEB-INF/文件夹名”,例如MyController.java里的mv.setViewName(“…….”)

这样一来,再直接访问show.jsp就会报404

6. 视图解析器

6.1 作用

当我们在view文件夹里面,再建一个视图文件,例如other.jsp

想用这个新的视图文件实现和原先相同的功能,只需要作出如下更改:

将MyController.java里面的指定视图改为:

mv.setViewName(“/WEB-INF/view/other.jsp”);

====>这两个视图路径只有视图文件名发生变化,而其他没有变化

框架一看,有重复工作==>我帮你做:视图解析器(解决视图路径重复问题)

视图解析器可选,不用也行

6.2 使用方法

  1. 在配置文件中声明视图解析器(下面这个是众多的视图解析器中的一种)

    <?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xsi:schemaLocation="http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans.xsd http://www.springframework.org/schema/context https://www.springframework.org/schema/context/spring-context.xsd">
    
<!--springmvc配置文件-->
<!--声明组件扫描器-->
    <context:component-scan base-package="com.bjpowernode.controller"/>
<!--    声明视图解析器,帮助处理视图 -->
   <bean class="org.springframework.web.servlet.view.InternalResourceViewResolver">
       <!--        前缀:指定视图文件的路径-->
       <property name="prefix" value="/WEB-INF/view/"/>
       <!--        后缀:视图文件的拓展名-->
       <property name="suffix" value=".jsp"/>
   </bean>
</beans>

  2. MyController中指定视图setViewName的修改: mv.setViewName(“文件名”);

    • 当配置了视图解析器,使用文件名作为视图名使用,叫做视图逻辑名称
    • 当使用了逻辑名称,框架使用配置文件中视图解析器的前缀,后缀和逻辑名称,拼接为完整视图路径,本例: /WEB-INF/view/show.jsp
    @Controller
public class MyController  {
    @RequestMapping(value = "/some.do")
    public ModelAndView dosome(){
        //调用service对象,处理请求,返回数据
        ModelAndView mv=new ModelAndView();
        //添加数据
        mv.addObject("msg", "处理了some.do请求");
        mv.addObject("fun","执行了dosome方法");
        // 当配置了视图解析器,使用文件名作为视图名使用,叫做视图逻辑名称
        //当使用了逻辑名称,框架使用配置文件中视图解析器的前缀,后缀和逻辑名称,拼接为完整视图路径
        //    /WEB-INF/view/show.jsp
        mv.setViewName("show");
        //返回结果
        return  mv;
    }
}

9.4 SpringMVC对请求方式的定义

对于@RequestMapping,具有一个属性method,用于对被注解方法所处理请求的提交方式进行限制,即只有满足该method属性指定的提交方式的请求,才会执行该注解的方法

@RequestMapping(value = "some.do",method = RequestMethod.GET )

​ ctrl+鼠标点进RequestMethod:

Method属性的取值为RequestMethod枚举常量,常用的为RequestMethod.GET与RequestMethod.POST,分别表示提交方式的匹配规则为GET与POST提交
get与post区别:

  • get传输的数据比较小

  • get在浏览器地址栏上显示数据,不够安全

  • get可以用浏览器进行缓存处理(浏览器能够缓存get请求的数据)

  • 相比于post,get是每次做请求时,都是重新提交请求,从服务器获取最新的结果

  • post没有缓存功能

  • 对数据添加修改一般都是post

  • 查询的时候用get多一些

  • 参数少的时候用get多些

  • 数据经常不变化的用get多

客户端浏览器常用的请求方式,及其提交方式有以下几种:

请求方式 提交方式
表单请求 默认GET,可以指定POST
AJAX请求 默认GET,可以指定POST
地址栏请求 GET请求
超链接请求 GET请求
src资源路径请求 GET请求

也就是说,只要指定了处理器方法匹配的请求提交方式为POST,则相当于指定请求发送的方式:要么使用表单请求,要么使用AJAX请求.其他请求方式被禁用

11.4.1 HttpMessageConverter接收请求数据转为对象

HttpMessageConverter(消息转换器) 接口:

作用:

  • 实现请求的数据转换为java对象
  • 把控制器方法返回的对象转为json,xml,text,二进制等不同格式的数据

① canRead()和read()方法的说明

  1. 在idea里面shift+shift,然后输入HttpMessageConverter,找到该类,点进去
  2. ctrl+O查看该类所有的方法
public interface HttpMessageConverter<T> {
    /**
    作用:检查clazz这个类型的对象,能否转为mediaType表示的数据格式
		 如果能转为mediaType表示的类型,返回true;
	      返回true则调用read()
    */
    boolean canRead(Class<?> var1, @Nullable MediaType var2);
    
    /**
    作用:接收请求中的数据,把数据转化为 calzz表示的对象
    */
     T read(Class<? extends T> var1, HttpInputMessage var2) throws IOException, HttpMessageNotReadableException;

    /**
    作用:检查calzz这种数据类型,能否转为mediaType表示的数据格式
    */
    boolean canWrite(Class<?> var1, @Nullable MediaType var2);

    
    /**
    作用:把t对象,按照contextType说明的格式,把对象转为json或者xml   t就是控制器方法的返回值
    */
    void write(T var1, @Nullable MediaType var2, HttpOutputMessage var3) throws IOException, HttpMessageNotWritableException;
    
    List<MediaType> getSupportedMediaTypes();
}

以上这些方法都不需要自己手动去做,是框架根据类型,自动的来判断处理的

② MediaType的说明

MediaType(媒体类型):表示在互联网中,应用程序之间在进行数据交换时,用的数据格式.用MediaType表示数据的格式

11.4.2 HttpMessageConverter的实现类


为什么有这么多实现类?

主要是因为框架不仅需要支持json格式,也可能需要用到html格式或者二进制格式等等;所以需要有更多不同的实现类来处理不同的数据我们只需要知道两个就可以了

  • MappingJackson2HttpMessageConverter
    用jackson工具库的objectMapper把java对象转为json数据格式
  • StringHttpMessageConverter
    把字符串类型的数据,进行格式转换和编码

① 注解驱动标签

怎么使用实现类:

框架根据控制器方法的返回值类型,自动查找使用的实现类

在程序中拷贝一个控制器方法

@RequestMapping(value = "return-void-ajax.do")
public Student doReternStringView1 (String name,Integer age) {
    Student student=new Student();
    student.setAge(12);
    student.setName("gxxt");
    return student;
}

现在这个代码返回的对象是Student,框架根据返回值类型,自动去查找能够处理这个类型的实现类

在默认的情况下,springmvc使用了HttpMessageConverter接口的四个实现类,其中包括了StringHttpMessageConverter(注意:不包括MappingJackson2HttpMessageConverter),这也就意味着,默认情况下,不能将一个返回值类型转换为json==>需要让程序找到MappingJackson2HttpMessageConverter来完成转换

===>需要在springmvc的配置文件中,加入注解驱动的标签mvc:annotation-driven. 加入这个标签之后,springmvc项目启动后,会创建HttpMessageConverter接口的七个实现类,包括:StringHttpMessageConverter和MappingJackson2HttpMessageConverter
